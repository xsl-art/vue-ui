# vue-ui 项目面试问答

> 一题一答，覆盖组件库设计、工程化、典型实现与细节。答案基于当前仓库实现，可按实际参与程度裁剪表述。

---

## Q1：请用 1～2 分钟介绍这个项目。

**答：** 这是一个基于 Vue 3 和 TypeScript 的轻量级组件库，用 Vite 以 library 模式打包成 ES 和 UMD 两种格式，并单独输出 CSS 和类型声明。包含 Button、Input、Select、Form 校验、Dialog、Tooltip、Dropdown、Message 等常用组件。工程上配套 Vitest 单测和 VitePress 文档站。比较有代表性的实现包括：Form 用 provide/inject 收集表单项并用 async-validator 校验；Message 用 `h` + `render` 做命令式挂载；Tooltip 用 Popper.js 做定位；Dialog 用 Teleport 和引用计数的 body 滚动锁定。

---

## Q2：为什么要自己做组件库，而不是直接用 Element Plus？

**答：** 业务项目可以直接用成熟 UI 库。自己做组件库的价值在于深入理解组件 API 设计、表单校验链路、浮层定位、库打包与类型发布等底层能力，适合作为作品集或面试项目。本库定位轻量、可读完源码；生产环境若追求完整性与无障碍，仍建议用 Element Plus 等成熟方案，或在本库之上只做二次封装。

---

## Q3：组件库是如何打包并供外部项目使用的？

**答：** `vite.config.ts` 配置 `build.lib`，入口是 `src/index.ts`，输出 ES（`dist/es/index.mjs`）和 UMD（`dist/umd/index.umd.js`）。`rollupOptions.external` 把 `vue` 排除在 bundle 外，作为 `peerDependencies`。样式打包为 `dist/style.css`，`package.json` 的 `exports` 暴露主入口和 `./style.css`。类型由 `vue-tsc -p tsconfig.build.json` 生成到 `dist/types`。使用者 `app.use(VueUi)` 全局注册，或按需 `import { Button } from 'vue-ui'` 并单独引入样式。

---

## Q4：为什么要把 Vue 设为 external，而不是打进组件库？

**答：** Vue 是单例运行时，若打进库会导致业务项目存在两份 Vue，破坏响应式与插件系统。标准做法是把 Vue 列为 `peerDependencies`，由宿主应用提供唯一 Vue 实例。UMD 场景通过 `output.globals` 映射到全局 `Vue`。

---

## Q5：`install` 方法是如何实现全局注册的？

**答：** `src/index.ts` 里定义 `components` 数组，包含所有带 `name` 的组件（如 `VkButton`）。`install(app)` 遍历并调用 `app.component(component.name!, component)`。默认导出 `{ install }`，与 `app.use()` 约定一致。Font Awesome 图标库在入口 `library.add(fas)` 统一注册，避免每个 Icon 单独引入。

---

## Q6：Form 和 FormItem 之间如何通信？为什么不用 props 层层传递？

**答：** 使用 `provide/inject`。`Form` provide `formContextKey`，包含 `model`、`rules`、`addField`、`removeField` 等。`FormItem` 在 mounted 时把自己注册进 Form 的 `fields` 数组，unmounted 时移除。这样 FormItem 可以嵌套在任意深度的 slot 里，Input 再 inject `formItemContextKey` 触发字段级校验。比逐层 props 更符合表单组件树的实际 DOM 结构。

---

## Q7：表单校验用的是什么库？整体 `validate` 流程是怎样的？

**答：** 使用 `async-validator`。每个 `FormItem` 根据 `prop` 从 `rules` 取规则，按 `trigger`（blur、change、input）过滤后构造 `Schema` 实例校验。`Form.validate()` 顺序 `await field.validate('')`，收集所有失败字段合并为 `ValidateFieldsError`，无错误返回 `true`，否则 `Promise.reject`。Input 在 blur、input、change 时调用 `formItemContext.validate(trigger)`，实现交互式校验。

---

## Q8：Input 的 v-model 是如何实现的？和 Form 如何联动？

**答：** 内部 `innerValue` ref 绑定原生 input，`watch` 外部 `modelValue` 同步。`handleInput` 时 `emit('update:modelValue')` 和 `emit('input')`，并 `runValidate('input')`。通过 `inject(formItemContextKey)` 拿到 FormItem 的 `inputId` 赋给原生 `id`，关联 label 的 `for` 与错误区域的 `aria` 属性。`inheritAttrs: false` 便于把 `$attrs` 精确绑到原生元素。

---

## Q9：Tooltip 的定位原理是什么？

**答：** 打开浮层后，在 `watch(isOpen)` 且 `flush: 'post'` 阶段，对 trigger 节点和 popper 节点调用 `@popperjs/core` 的 `createPopper`，传入 `placement` 和用户可覆盖的 `popperOptions`（含 offset modifier）。关闭或节点缺失时 `destroy` 实例。这比手写 getBoundingClientRect 更稳，且支持翻转、边界检测等 modifier 扩展。

---

## Q10：Dropdown 为什么基于 Tooltip 而不是单独写一套浮层？

**答：** Dropdown 本质是「触发器 + 弹出菜单」，与 Tooltip 的显隐、定位、clickOutside、manual 控制一致。复用 `VkTooltip` 减少重复代码，只在 `#content` 插槽渲染 `menuOptions` 列表，并暴露 `show/hide` 转调 Tooltip。菜单项 `label` 支持 VNode，通过 `RenderVNode` 渲染，便于自定义图标或复杂内容。

---

## Q11：`useClickOutside` 是怎么实现的？有什么注意点？

**答：** 在 `onMounted` 给 `document` 绑定 `click` 监听，判断 `event.target` 是否在 `elementRef` 容器内（`contains`）。若在外部则执行回调，Tooltip 在 click 模式下用于关闭浮层。注意：document 级监听要在 `onUnmounted` 移除；与 trigger 的 click 打开可能存在时序问题，复杂场景有时需要 `mousedown` 或延迟判断。

---

## Q12：Message 为什么用命令式 API？实现步骤是什么？

**答：** 提示类组件常在任意逻辑处弹出（如请求失败），用 `createMessage({ message, type })` 比每次在模板里放 `<Message>` 更方便。实现：`document.createElement('div')` → `h(Message, props)` → `render(vnode, container)` → 把 `firstChild` append 到 `body`。维护 `instances` 数组做堆叠偏移和 `closeAll`。销毁时 `render(null, container)` 卸载 VNode 并移除 DOM。这是 Vue 3 无 Vue 2 `Vue.extend` 时的标准 imperative 写法。

---

## Q13：多条 Message 如何垂直堆叠不重叠？

**答：** `getLastBottomOffset(id)` 查找当前实例在数组中的前一个实例，读取其暴露的 `bottomOffset`（上一条的 top + 高度）。当前 Message 的 `top = offset + lastOffset`，在 `Transition` enter 后 `updateHeight` 测量自身高度写回 `bottomOffset` 供下一条使用。这样每条消息的 `style.top` 动态累加。

---

## Q14：Dialog 打开时如何防止背景页面滚动？

**答：** `scrollLock.ts` 实现引用计数 `scrollLockCount`。第一次锁定记录 `scrollTop`，把 `body` 设为 `position: fixed; top: -scrollTop`，并 `overflow: hidden`。关闭时计数减一，为 0 时恢复原有 style 并 `window.scrollTo(0, scrollTop)`。多个 Dialog 连续打开时不会提前解锁，避免滚动条闪烁和位置跳动。

---

## Q15：Dialog 的 `Teleport` 和 `destroyOnClose` 分别解决什么问题？

**答：** `Teleport to="body"` 把遮罩和对话框挂到 body 下，避免父级 `overflow: hidden` 或 `transform` 影响 fixed 定位与 z-index 层叠。`destroyOnClose` 在关闭动画结束后把 `bodyMounted` 设为 false，卸载默认插槽内容，释放大表单或重资源子树，下次打开重新挂载。

---

## Q16：全局 z-index 如何管理？为什么需要 `useZIndex`？

**答：** 模块内共享 `ref(0)`，每次 `nextZIndex()` 自增，返回 `initialValue + 当前计数`。Dialog 打开、Message 创建时各取一层，保证后打开的浮层在上。若各组件写死 z-index，多个 Dialog 与 Message 同时存在时会互相遮挡。

---

## Q17：Collapse 手风琴模式如何实现？

**答：** `accordion` 为 true 时，`handleClick` 若点击已展开项则 `activeNames` 置空，否则只保留当前项 `[item]`。多选模式则在数组中 toggle 增删。状态通过 `provide` 传给 `CollapseItem`，子项只负责展示和触发 `handleClick`。`watch modelValue` 与外部 v-model 同步。

---

## Q18：`RenderVNode` 组件的作用是什么？

**答：** 部分 API 需要支持渲染函数或 VNode 作为内容（如 Select 的 `renderLabel`、Dropdown 的 `label`、Message 的 `message`）。`RenderVNode` 接收 `VNode` prop，在 setup 里 `return () => props.VNode` 作为渲染函数输出。比模板里直接插 VNode 更统一，也避免在多个组件重复写渲染逻辑。

---

## Q19：Select 组件复杂在哪里？依赖了哪些子组件？

**答：** Select 组合了 Input（输入与 v-model）、Tooltip（下拉浮层 manual 控制）、Tag（多选标签）、Icon、RenderVNode。还要处理：单选/多选、filterable 防抖搜索、键盘上下与 Enter、ARIA combobox/listbox、选项高亮与清空。是库中集成度最高的组件之一，阅读建议先看 Tooltip 和 Input 再读 Select。

---

## Q20：组件库如何做单元测试？

**答：** Vitest + jsdom + `@vue/test-utils`。`vitest.config.ts` 指定 `setupFiles` 和 coverage 范围。例如 `form.spec.ts` 用 `defineComponent` 包一层 Host，内嵌 Form、FormItem、Input，stub 掉 Icon，测试 `validate`、`resetFields`、label 与 `inputId` 绑定等。运行 `pnpm test` 或 `pnpm test:coverage` 查看覆盖。

---

## Q21：TypeScript 类型如何对外暴露？

**答：** 各组件 `types.ts` 定义 Props、Emits、Instance 接口。`index.ts` 使用 `export type * from './components/xxx'` 聚合导出。构建时 `vue-tsc` 根据 Vue SFC 生成 `dist/types/index.d.ts`，`package.json` 的 `"types"` 指向该文件。业务侧可获得 `FormInstance`、`ButtonProps` 等智能提示。

---

## Q22：样式如何组织？如何支持主题扩展？

**答：** 全局 `styles/_var.scss` 定义 CSS 变量（颜色、圆角、字体等），各组件 SCSS 使用 `var(--vk-color-primary)` 等。发布时样式打进 `style.css`，业务项目引入一次即可。主题扩展可覆盖 `:root` 变量，无需改组件源码；未做运行时主题切换 API。

---

## Q23：Popper 的 `openDelay` / `closeDelay` 为什么要 debounce 且互斥 cancel？

**答：** hover 时鼠标快速划过不应立刻闪烁开关。debounce 延迟执行 open/close。从 trigger 移到 popper 时，会先触发 leave 的 closeDebounced，若不同步 cancel 会导致浮层误关。因此在 `openFinal` 里 `closeDebounce.cancel()` 再 `openDebounce()`，反之亦然，保证 hover 链路连贯。

---

## Q24：组件命名为什么用 `Vk` 前缀？

**答：** `defineOptions({ name: 'VkForm' })` 等，对应样式类 `vk-form`，避免与业务组件或其他库同名冲突，也便于在 DevTools 中识别库组件。全局注册后模板里使用 `<VkButton>` 或取决于是否配置了 resolver；文档示例中常写组件标签名与 name 一致。

---

## Q25：peerDependencies 和 dependencies 如何划分？

**答：** `vue` 放 `peerDependencies`，由宿主提供。`async-validator`、`lodash-es`、`@popperjs/core`、Font Awesome 等打在库 bundle 或 dependencies 中，随 npm 包安装。注意 lodash 全量包也在 dependencies 中，若体积敏感可只保留 `lodash-es` 并按函数 tree-shaking。

---

## Q26：VitePress 文档和库开发是什么关系？

**答：** `docs/` 是独立文档站，`pnpm docs:dev` 启动。通过 `@vitepress-demo-preview` 嵌入 `docs/demo` 下的示例 Vue 文件，用于展示 props 与交互。库本身 `pnpm dev` 的 `App.vue` 几乎为空，真实演示以文档站为主，这是许多组件库的常见结构。

---

## Q27：若业务项目只想用 Button 和 Input，如何按需引入？

**答：** `import { Button, Input } from 'vue-ui'` 和 `import 'vue-ui/style.css'`。在组件的 `components` 选项注册即可，无需 `app.use` 全量 install。注意：未使用 unplugin 自动按需时，CSS 仍是全量 style.css；进一步可做按组件拆 CSS（当前未实现）。

---

## Q28：Form 的 `resetFields` 和 `clearValidate` 区别？

**答：** `resetFields` 把 model 中对应字段恢复为 FormItem 挂载时记录的 `initialValue`，并清除校验状态。`clearValidate` 只清除错误提示和 loading，不改 model 值。两者都可传字段名数组，不传则作用于全部已注册 field。

---

## Q29：Dialog 拖拽如何避免拖出视口？

**答：** mousedown 时记录 dialog 的 `getBoundingClientRect` 和当前偏移，mousemove 时计算新偏移并用 `minX/maxX/minY/maxY` clamp，边界基于视口宽高与 dialog 当前位置。关闭按钮区域不启动拖拽（`closest('.vk-dialog__close')`）。

---

## Q30：如果让你继续完善这个库，优先做哪三件事？

**答：** 1）补全 dev  playground 或 Storybook，方便不启文档站也能调试；2）Select / Form 拆 composable + 补键盘与 a11y 测试用例；3）构建侧增加按需样式、changelog 与 CI（test + build + 发布）。可选：用 `unplugin-vue-components` 做 resolver、修正 `commen` 目录拼写。

---

## Q31：这个项目里你负责/最熟悉的模块是什么？（模板，需自行替换）

**答：** （示例）我主要负责 Form 链路和 Message 命令式 API。Form 侧完成了 provide/inject 注册、async-validator 与 Input 触发器联动；Message 侧处理了多实例堆叠偏移和销毁逻辑。难点是 z-index 与 body 滚动锁在 Dialog + Message 同时打开时的一致性。

---

## Q32：`defineExpose` 在组件库中的典型用法？

**答：** Form 暴露 `validate`、`resetFields`；Dialog 暴露 `open`、`close`；Tooltip 暴露 `show`、`hide`、`update`（更新 Popper 位置）。业务通过 `ref` 调用 imperative 方法。Message 实例则通过 `createMessage` 返回对象的 `destory` 方法关闭。
