300ms:  请求 B 返回 ["Apple", "Abacus"]
300ms:  filteredOptions = ["Apple", "Abacus"] ✅ 正确
500ms:  请求 A 返回 ["Apple", "Ant", "Ask"]
500ms:  filteredOptions = ["Apple", "Ant", "Ask"] ❌ 错误！覆盖了 B 的结果
用户看到的结果是 "a" 的搜索，但输入框显示的是 "ab"
```

**防覆盖方案：**

**方案一：请求令牌（推荐）**
```ts
let currentRequestId = 0;

const generateFilterOptions = async (searchValue: string) => {
  if (props.remote && props.remoteMethod) {
    const requestId = ++currentRequestId; // 递增令牌
    states.loading = true;

    try {
      const results = await props.remoteMethod(searchValue);
      // 检查是否是最新请求
      if (requestId === currentRequestId) {
        filteredOptions.value = results;
      } else {
        console.log('忽略过期请求结果');
      }
    } catch (e) {
      if (requestId === currentRequestId) {
        filteredOptions.value = [];
      }
    } finally {
      if (requestId === currentRequestId) {
        states.loading = false;
      }
    }
  }
};
```

**方案二：AbortController 取消旧请求**
```ts
let abortController: AbortController | null = null;

const generateFilterOptions = async (searchValue: string) => {
  // 取消上一个请求
  abortController?.abort();
  abortController = new AbortController();

  if (props.remote && props.remoteMethod) {
    states.loading = true;
    try {
      filteredOptions.value = await props.remoteMethod(searchValue, {
        signal: abortController.signal,
      });
    } catch (e) {
      if (e.name !== 'AbortError') {
        filteredOptions.value = [];
      }
    } finally {
      states.loading = false;
    }
  }
};
```

**方案三：使用 lodash-es 的 debounce 取消特性**
```ts
const debounceOnFilter = debounce(onFilter, timeout.value);

// 在组件卸载时取消
onUnmounted(() => {
  debounceOnFilter.cancel();
});

// 但这只能取消未发出的请求，无法取消已发出的异步请求
```

**当前项目的 debounce 问题：**
- `debounce` 只能延迟执行，不能取消已发出的网络请求
- 需要在 `remoteMethod` 内部或外部包装取消逻辑

**推荐实现（结合 debounce + 请求令牌）：**
```ts
let currentRequestId = 0;

const fetchRemoteOptions = async (searchValue: string) => {
  const requestId = ++currentRequestId;
  states.loading = true;

  try {
    const results = await props.remoteMethod!(searchValue);
    if (requestId === currentRequestId && searchValue === states.inputValue) {
      filteredOptions.value = results;
    }
  } catch (e) {
    if (requestId === currentRequestId) {
      filteredOptions.value = [];
    }
  } finally {
    if (requestId === currentRequestId) {
      states.loading = false;
    }
  }
};

const debounceOnFilter = debounce(
  () => fetchRemoteOptions(states.inputValue),
  timeout.value,
);
```

### Q4：Popper 的 `flip`、`preventOverflow` 这些 modifier 配了吗？窗口 resize/scroll 时谁负责 `update()`？

**答：**

**当前 Popper 配置：**

```ts
// Select.vue
const sameWidthModifier: Modifier<"sameWidth", Record<string, never>> = {
  name: "sameWidth",
  enabled: true,
  fn: ({ state }) => {
    state.styles.popper.width = `${state.rects.reference.width}px`;
  },
  phase: "beforeWrite",
  requires: ["computeStyles"],
  effect: ({ state }) => {
    const reference = state.elements.reference;
    if (reference instanceof HTMLElement) {
      state.elements.popper.style.width = `${reference.offsetWidth}px`;
    }
  },
};

const popperOptions: Partial<Options> = {
  modifiers: [
    {
      name: "offset",
      options: {
        offset: [0, 4],
      },
    },
    sameWidthModifier,
  ],
};
```

**是否配置了 `flip`、`preventOverflow`？**
- **没有显式配置！**
- Popper.js 默认启用了以下 modifier：
  - `flip`（自动翻转）：✅ 默认启用
  - `preventOverflow`（防止溢出）：✅ 默认启用
  - `offset`（偏移）：✅ 显式配置
  - `sameWidth`（同步宽度）：✅ 自定义配置

**默认 modifier 的行为：**

| Modifier | 默认行为 | 当前配置 |
|----------|---------|---------|
| `flip` | 空间不足时自动翻转（如 bottom → top） | 默认启用，未自定义 |
| `preventOverflow` | 防止弹出层超出视口边界 | 默认启用，未自定义 |
| `hide` | 超出边界时添加 `data-popper-reference-hidden` 属性 | 默认启用 |
| `offset` | 设置弹出层与参考元素的距离 | 显式配置 `[0, 4]` |
| `sameWidth` | 同步弹出层和参考元素宽度 | 自定义实现 |

**窗口 resize/scroll 时谁负责 `update()`？**

**当前实现：**
- **没有监听 resize/scroll 事件！**
- 只在 Tooltip 的 `watch(isOpen)` 时创建 Popper 实例
- Popper 实例内部会自动监听窗口变化并调用 `update()`

**Popper.js 的自动更新机制：**
```ts
// Popper.js 源码内部
watch(update, [scroll, resize]);
// 默认监听 window 的 scroll 和 resize 事件
```

**但存在的问题：**

1. **滚动容器不是 window 时**：
   ```vue
   <div class="scrollable-container" style="overflow: auto; height: 500px;">
     <VkSelect /> <!-- 滚动这个容器时，Popper 不会更新 -->
   </div>
   ```
   - Popper 默认只监听 `window` 的滚动
   - 如果 Select 在可滚动的容器内，容器滚动时下拉框位置会错位

2. **手动更新调用有限**：
   ```ts
   // Select.vue
   const updateDropdownPosition = async () => {
     await nextTick();
     if (!isDropdownShow.value) return;
     toolTipRef.value?.update?.(); // 只在选择项后调用
   };

   watch(() => states.selectedOptions.length, () => {
     if (props.multiple) updateDropdownPosition();
   }, { flush: "post" });
   ```
   - 只在多选选项变化时更新，其他场景（如窗口缩放、动态内容）不更新

**改进方案：**

```ts
// Tooltip.vue
import { useResizeObserver } from '@vueuse/core';

// 监听 trigger 节点大小变化
useResizeObserver(triggrtNode, () => {
  if (isOpen.value) {
    popperInstance?.update();
  }
});

// 监听滚动容器
const setupScrollListeners = () => {
  const scrollParents = getScrollParents(triggrtNode.value);
  scrollParents.forEach(el => {
    el.addEventListener('scroll', popperInstance.update, { passive: true });
  });

  onUnmounted(() => {
    scrollParents.forEach(el => {
      el.removeEventListener('scroll', popperInstance.update);
    });
  });
};
```

### Q1：`scrollLock` 用引用计数，嵌套两个 Dialog 时 body 滚动锁对不对？关掉内层会不会误解锁？

**答：**

**当前实现（scrollLock.ts）：**

```ts
let scrollLockCount = 0;
let scrollTop = 0;
// ... 保存之前的样式

export function lockBodyScroll() {
  scrollLockCount++;
  if (scrollLockCount === 1) {
    // 只在第一次锁定时修改样式
    scrollTop = window.scrollY || document.documentElement.scrollTop;
    document.body.style.position = "fixed";
    document.body.style.top = `-${scrollTop}px`;
    document.body.style.width = "100%";
    document.body.style.overflow = "hidden";
  }
}

export function unlockBodyScroll() {
  scrollLockCount = Math.max(0, scrollLockCount - 1);
  if (scrollLockCount === 0) {
    // 只在计数归零时恢复样式
    document.body.style.position = previousBodyPosition;
    document.body.style.top = previousBodyTop;
    // ... 恢复其他样式
    window.scrollTo(0, scrollTop);
  }
}
```

**嵌套 Dialog 场景：**

```
打开 Dialog A
  ↓
lockBodyScroll() → scrollLockCount = 1 → 锁定滚动 ✅

打开 Dialog B（嵌套）
  ↓
lockBodyScroll() → scrollLockCount = 2 → 不重复锁定 ✅

关闭 Dialog B（内层）
  ↓
unlockBodyScroll() → scrollLockCount = 1 → 不解锁 ✅

关闭 Dialog A（外层）
  ↓
unlockBodyScroll() → scrollLockCount = 0 → 解锁滚动 ✅
```

**是否正确？**
- **基本正确！** 引用计数机制能处理嵌套场景
- 不会误解锁内层 Dialog

**但存在潜在问题：**

1. **Dialog 异常关闭未调用 unlock**：
   ```ts
   // Dialog.vue
   onBeforeUnmount(() => {
     if (props.modelValue && props.lockScroll) {
       unlockBodyScroll();
     }
   });
   ```
   - 如果 Dialog 在 `modelValue = true` 时被直接卸载（如 `v-if="false"`），会正确解锁
   - 但如果 `lockScroll` 属性动态变化，可能导致计数不匹配

2. **多个独立 Dialog 实例**：
   ```vue
   <Dialog v-model="showA" />
   <Dialog v-model="showB" />
   ```
   - 如果同时打开 A 和 B，计数 = 2
   - 关闭 A 后，计数 = 1，滚动仍锁定 ✅
   - 但如果 A 和 B 的 `lockScroll` 配置不同，可能混乱

3. **全局共享状态的副作用**：
   - `scrollLockCount` 是模块级变量，整个应用共享
   - 如果其他组件也调用 `lockBodyScroll`，可能干扰 Dialog

**改进建议：**

```ts
// 方案一：增加安全检查
export function unlockBodyScroll() {
  if (scrollLockCount === 0) {
    console.warn('[VueUi] unlockBodyScroll called without matching lock');
    return;
  }

  scrollLockCount--;
  if (scrollLockCount === 0) {
    // 恢复样式
  }
}

// 方案二：返回解锁函数（配对使用）
export function lockBodyScroll(): () => void {
  lockBodyScroll();
  return unlockBodyScroll;
}

// 使用
const unlock = lockBodyScroll();
// ...
unlock(); // 确保配对
```

**测试用例建议：**
```ts
it('should handle nested dialogs correctly', () => {
  lockBodyScroll();
  expect(scrollLockCount).toBe(1);
  expect(document.body.style.overflow).toBe('hidden');

  lockBodyScroll();
  expect(scrollLockCount).toBe(2);

  unlockBodyScroll();
  expect(scrollLockCount).toBe(1);
  expect(document.body.style.overflow).toBe('hidden'); // 仍锁定

  unlockBodyScroll();
  expect(scrollLockCount).toBe(0);
  expect(document.body.style.overflow).toBe(''); // 已解锁
});
```

### Q2：Teleport 到 `body` 后，z-index 和 Message 冲突怎么办？你的 `useZIndex` 是全局单例还是 per-component？

**答：**

**当前 z-index 管理机制：**

```ts
// useZIndex.ts
const DEFAULT_Z_INDEX_BASE = 2000;
let zIndexSeed = 0; // 全局共享种子

export function nextZIndex(base = DEFAULT_Z_INDEX_BASE): number {
  zIndexSeed += 1;
  return zIndexSeed + base;
}
```

**是全局单例还是 per-component？**
- **全局单例！** `zIndexSeed` 是模块级变量，整个应用共享
- 所有调用 `nextZIndex()` 的组件使用同一个递增计数器

**Dialog 和 Message 的 z-index 分配：**

```ts
// Dialog.vue
watch(() => props.modelValue, (newVal) => {
  if (newVal) {
    if (props.zIndex == null) {
      currentZIndex.value = nextZIndex(2000); // 从 2001 开始
    }
  }
});

// Message methods.ts
const newProps = {
  ...props,
  zIndex: nextZIndex(), // 也是从 2001 开始
};
```

**冲突场景：**

```
时间线:
T1: 打开 Dialog A → zIndex = 2001
T2: 显示 Message → zIndex = 2002
T3: 打开 Dialog B → zIndex = 2003

DOM 结构:
<body>
  <div class="vk-dialog" style="z-index: 2001">Dialog A</div>
  <div class="vk-message" style="z-index: 2002">Message</div>
  <div class="vk-dialog" style="z-index: 2003">Dialog B</div>
</body>

结果: Message (2002) 在 Dialog A (2001) 上方 ✅
      Dialog B (2003) 在最上方 ✅
```

**一般不会冲突，但有潜在问题：**

1. **固定 base 值的风险**：
   ```ts
   // Dialog 使用 base = 2000
   currentZIndex.value = nextZIndex(2000);

   // Message 使用默认 base = 2000
   zIndex: nextZIndex();

   // 如果用户自定义 base
   nextZIndex(3000); // 跳到 3001+，可能导致 Dialog 被 Message 覆盖
   ```

2. **z-index 不会回收**：
   ```
   关闭 Dialog A 后，2001 这个 z-index 不会被回收
   新的组件继续从 2004、2005 递增
   长时间运行可能导致 z-index 过大（虽然实际上不会有问题）
   ```

3. **Teleport 到 body 后的层叠上下文**：
   ```vue
   <Teleport to="body">
     <div class="vk-dialog" style="z-index: 2001">
       <!-- 如果内部有固定定位元素，可能创建新的层叠上下文 -->
     </div>
   </Teleport>
   ```

**改进方案：**

**方案一：分层管理（推荐）**
```ts
// useZIndex.ts
const Z_INDEX_LAYERS = {
  tooltip: 1000,
  dropdown: 1500,
  message: 2000,
  dialog: 2500,
  modal: 3000,
};

const layerSeeds = {
  tooltip: 0,
  dropdown: 0,
  message: 0,
  dialog: 0,
  modal: 0,
};

export function nextZIndex(layer: keyof typeof Z_INDEX_LAYERS) {
  layerSeeds[layer]++;
  return Z_INDEX_LAYERS[layer] + layerSeeds[layer];
}

// 使用
Dialog: nextZIndex('dialog');    // 2501, 2502, ...
Message: nextZIndex('message');  // 2001, 2002, ...
```

**方案二：提供配置项**
```ts
// 允许用户自定义 z-index 起始值
export function nextZIndex(base = DEFAULT_Z_INDEX_BASE): number {
  // 支持手动指定
}

// Dialog.vue
<Dialog :z-index="5000" /> // 用户可覆盖
```

### Q3：拖拽 Dialog 时，如果超出视口，边界约束做了吗？

**答：**

**当前拖拽实现：**

```ts
// Dialog.vue
const onHeaderMouseDown = (e: MouseEvent) => {
  if (!props.draggable || props.fullscreen) return;
  if (e.button !== 0) return;

  const dialogEl = dialogRef.value;
  if (!dialogEl) return;

  e.preventDefault();
  dragging.value = true;

  const rect = dialogEl.getBoundingClientRect();
  const startX = e.clientX;
  const startY = e.clientY;
  const startOffset = { ...dialogOffset.value };

  // 边界约束计算
  const minX = -rect.left + startOffset.x;
  const maxX = document.documentElement.clientWidth - rect.right + startOffset.x;
  const minY = -rect.top + startOffset.y;
  const maxY = document.documentElement.clientHeight - rect.bottom + startOffset.y;

  const onMouseMove = (moveEvent: MouseEvent) => {
    if (!dragging.value) return;
    dialogOffset.value = {
      x: Math.min(Math.max(startOffset.x + moveEvent.clientX - startX, minX), maxX),
      y: Math.min(Math.max(startOffset.y + moveEvent.clientY - startY, minY), maxY),
    };
  };

  const onMouseUp = () => {
    dragging.value = false;
    document.removeEventListener("mousemove", onMouseMove);
    document.removeEventListener("mouseup", onMouseUp);
  };

  document.addEventListener("mousemove", onMouseMove);
  document.addEventListener("mouseup", onMouseUp);
};
```

**是否做了边界约束？**
- **做了！** 通过计算 `minX`、`maxX`、`minY`、`maxY` 限制拖拽范围

**约束逻辑解析：**

```ts
// 假设 Dialog 宽度 500px，视口宽度 1920px，初始居中
const rect = { left: 710, right: 1210, top: 100, bottom: 600 };

// 边界计算
const minX = -710 + 0 = -710;  // Dialog 左边缘最多移出视口 710px（即完全贴左边界）
const maxX = 1920 - 1210 + 0 = 710; // Dialog 右边缘最多移出视口 710px（即完全贴右边界）
const minY = -100 + 0 = -100;  // Dialog 顶部最多贴顶
const maxY = 1080 - 600 + 0 = 480; // Dialog 底部最多贴底

// 拖拽时
dialogOffset.value = {
  x: Math.min(Math.max(newX, -710), 710), // 限制在 [-710, 710]
  y: Math.min(Math.max(newY, -100), 480), // 限制在 [-100, 480]
};
```

**约束效果：**
- ✅ Dialog 不能完全拖出视口
- ✅ 至少保持一部分在视口内
- ✅ 不会拖到负坐标导致消失

**但仍存在的问题：**

1. **滚动后视口变化**：
   ```ts
   // 计算使用的是 document.documentElement.clientWidth
   // 如果页面有滚动，clientHeight 可能不准确
   const maxY = document.documentElement.clientHeight - rect.bottom + startOffset.y;
   ```
   - 如果用户拖拽过程中页面滚动，边界可能计算错误

2. **多显示器场景**：
   - `clientWidth` 只计算主显示器
   - 如果浏览器窗口跨显示器，边界约束可能不准确

3. **动态调整窗口大小**：
   ```ts
   // 边界在 mousedown 时计算一次
   // 如果拖拽过程中用户调整浏览器窗口大小，边界不会更新
   ```

**改进建议：**

```ts
const onMouseMove = (moveEvent: MouseEvent) => {
  // 每次移动时重新计算边界（适配窗口大小变化）
  const currentRect = dialogEl!.getBoundingClientRect();
  const currentMinX = -currentRect.left + dialogOffset.value.x;
  const currentMaxX = document.documentElement.clientWidth - currentRect.right + dialogOffset.value.x;

  dialogOffset.value = {
    x: Math.min(Math.max(startOffset.x + moveEvent.clientX - startX, currentMinX), currentMaxX),
    y: Math.min(Math.max(startOffset.y + moveEvent.clientY - startY, currentMinY), currentMaxY),
  };
};
```

**或者增加安全边距：**
```ts
const SAFE_MARGIN = 50; // 至少保留 50px 在视口内
const minX = -rect.left + startOffset.x + SAFE_MARGIN;
const maxX = document.documentElement.clientWidth - rect.right + startOffset.x - SAFE_MARGIN;
```

**可能的冲突场景：**

1. **浏览器全屏模式**：
   - 按 ESC 退出全屏
   - Dialog 也响应 ESC 关闭
   - **结果**：两者都触发，可能不符合预期

2. **表单自动补全/下拉建议**：
   ```vue
   <Dialog>
     <input type="text" /> <!-- 浏览器显示自动补全建议 -->
   </Dialog>
   ```
   - 按 ESC 关闭浏览器建议列表
   - Dialog 也响应 ESC 关闭
   - **结果**：Dialog 被意外关闭

3. **媒体播放控制**：
   - 某些浏览器/插件用 ESC 暂停视频
   - Dialog 同时关闭

**当前实现的保护机制：**

1. **只在 Dialog 打开时响应**：
   ```ts
   if (!props.modelValue) return;
   ```
   - 避免 Dialog 关闭后仍监听 ESC

2. **document 级监听**：
   - 使用 `useEventListener(document, 'keydown', onEsc)`
   - 会捕获所有 ESC 按键，无法区分目标元素

**问题示例：**

```vue
<Dialog v-model="show">
  <input ref="inputRef" />
</Dialog>

<!-- 用户操作流程 -->
1. 打开 Dialog，焦点在 input
2. 浏览器显示拼写检查/自动补全
3. 用户按 ESC 想关闭浏览器提示
4. Dialog 也关闭了 ❌
```

**改进方案：**

**方案一：检查事件目标**
```ts
const onEsc = (e: Event) => {
  const ev = e as KeyboardEvent;
  if (ev.key !== "Escape") return;
  if (!props.modelValue) return;

  // 如果焦点在输入框且有浏览器 UI 显示，不关闭 Dialog
  const target = ev.target as HTMLElement;
  if (target.tagName === 'INPUT' || target.tagName === 'TEXTAREA') {
    // 可选：检查是否有浏览器 UI 显示（难以检测）
    // 简单做法：不阻止，让事件冒泡，浏览器先处理
  }

  handleClose();
};
```

**方案二：使用 capture 阶段并 stopPropagation**
```ts
// 在捕获阶段处理，优先于浏览器默认行为
useEventListener(document, 'keydown', onEsc, { capture: true });

const onEsc = (e: Event) => {
  const ev = e as KeyboardEvent;
  if (ev.key !== "Escape") return;
  if (!props.modelValue) return;

  e.stopPropagation(); // 阻止事件继续传播
  e.preventDefault(); // 阻止浏览器默认行为
  handleClose();
};
```

**方案三：提供配置项（推荐）**
```vue
<Dialog
  v-model="show"
  :close-on-press-escape="false" <!-- 禁用 ESC 关闭 -->
/>

<!-- 或者 -->
<Dialog
  v-model="show"
  @keydown.esc="customEscHandler" <!-- 自定义 ESC 处理 -->
/>
```

**方案四：焦点管理（最佳实践）**
```ts
// Dialog 打开时，将焦点移到 Dialog 内部
watch(() => props.modelValue, (newVal) => {
  if (newVal) {
    nextTick(() => {
      dialogRef.value?.focus(); // 焦点在 Dialog 容器上
    });
  }
});

// ESC 只在全局焦点在 Dialog 内时关闭
const onEsc = (e: Event) => {
  if (ev.key !== "Escape") return;
  if (!props.modelValue) return;

  // 检查当前焦点是否在 Dialog 内
  if (!dialogRef.value?.contains(document.activeElement)) {
    return; // 焦点不在 Dialog 内，不关闭
  }

  handleClose();
};
```

**推荐做法：**
- 默认允许 ESC 关闭（符合用户预期）
- 提供 `closeOnPressEscape` 属性允许禁用
- 在文档中说明 ESC 的行为和可能的冲突
- 对于有输入框的场景，考虑延迟关闭或二次确认

### Q3：VitePress Demo 和真实打包产物行为不一致时，你怎么排查？

**答：**

**常见不一致原因：**

1. **源码路径差异**：
   ```
   VitePress Demo: 直接引用 src/components/Button.vue
   打包产物: 使用 dist/es/components/button/index.mjs
   ```

2. **构建配置差异**：
   - VitePress 使用 Vite 开发服务器（未压缩、未 tree-shaking）
   - 打包产物经过 Rollup 优化、压缩、external 处理

3. **样式加载差异**：
   - VitePress: SCSS 即时编译
   - 打包产物: 预编译的 CSS 文件

4. **依赖版本差异**：
   - VitePress: 使用 `node_modules` 中的依赖
   - 打包产物: 依赖可能被打进包或 external

**排查步骤：**
